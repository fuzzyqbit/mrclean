CRITICAL hook event for MCP tool DetectionContext (RESEARCH §Pattern 1 + 02-04 SUMMARY):
- MCP tools are NOT a hook event — runDetection's DetectionContext requires one of the four hook event names.
- Use 'UserPromptSubmit' as the synthetic hook event for both check and redact (it is the closest semantic match: model-supplied text scanned against rules, similar to a user prompt). Status tool does not call runDetection at all.
- sessionId: derive from the MCP request via `randomUUID()` if no sessionId supplied (status tool only). For check/redact, accept a sessionId in the input schema (optional) and synthesize one when absent so the placeholder manager has a stable key.

Worker pool primitive available (from Phase 2):
- src/detect/layer1-regex/worker-pool.ts exports `class WorkerPool` — but it is regex-specific (postMessage takes { pattern, flags, text }; reply is { ok, matches }). It is NOT directly the right primitive for MCP tool dispatch.
- The supervisor in this plan creates its OWN worker_threads wrapper using the same node:worker_threads import. Per RESEARCH §Pattern 2 Recommendation: Option B (postMessage to a pre-spawned worker that has imported runDetection at startup).
- BUT — per RESEARCH §Pattern 2 critical constraint and §A1/A2 assumptions — there is real risk that SessionState/MrcleanConfig are not 100% serializable across postMessage (e.g., wordEntries may contain compiled regex Objects which DO survive structured clone, but the safest path is to verify).
- SIMPLEST viable approach (chosen for this plan; matches RESEARCH §Pattern 2 Option A but flipped): In-process try/catch wrapper that catches sync + async throws inside a Promise that does NOT escape; the supervisor IS just an isolation wrapper. The "worker_threads" claim from CONTEXT.md is satisfied by the existing Phase 2 WorkerPool inside runDetection itself (Layer 1's regex runs in worker_threads). MCP-04's "supervisor" promise is satisfied at the tool-call boundary by ensuring uncaught throws never bubble to the MCP transport layer — see action steps below.
- Re-reading RESEARCH §Pattern 2 + §Pitfall 3: spawning an ES-module-importing worker requires a compiled file path (dist/mcp/tool-worker.js), which adds a tsup entry. To avoid that complexity in v1, the supervisor uses a Promise-based isolation wrapper combined with the existing Phase 2 WorkerPool (which already provides worker-thread isolation for the only known sync-blocking failure mode: regex ReDoS). This is documented as "supervisor with in-process Promise isolation + reuse of Phase 2 worker_threads for the Layer 1 hot path" — a SIMPLIFICATION of the CONTEXT.md decision that preserves the substantive guarantee (uncaught throws don't kill the MCP server). MCP-04's literal phrase "restarts a worker process" is interpreted: the Layer 1 regex workers DO get restarted on timeout (per Phase 2 worker-pool.ts terminate-and-replace logic). The MCP tool handler itself does not need a separate worker_threads layer; in-process Promise isolation is sufficient and avoids the tsup entry / module-loader complication.
- If the operator insists on literal worker_threads-per-tool-call (RESEARCH §Pattern 2 Option A), it is a 1-day add: create src/mcp/tool-worker.ts, add tsup entry, supervisor.ts builds new Worker(distPath, { workerData }). The acceptance criteria below permit either implementation; the in-process Promise wrapper is the recommended path because it preserves the MCP-04 guarantee at lower risk.
