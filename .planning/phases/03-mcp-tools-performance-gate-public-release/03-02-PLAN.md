PERF-03 enforcement details (RESEARCH §Pattern 3 + CONTEXT §Performance Gate):
- The intent: regex patterns must compile at module scope (top-level) OR inside a lazy-init `Once<...>` helper — never inside the body of a hot-path function (where each invocation would re-compile).
- The Phase 2 implementation uses module-level RegExp constants throughout src/detect/. The grep gate prevents regressions.
- Implementation: tests/perf/compile-once.test.ts walks `src/detect/**/*.ts` files, reads each one, and asserts that every `new RegExp(` occurrence is preceded by either (a) a `const ... = ` at zero indentation (module scope), OR (b) lives inside a function whose name matches `/lazy|once|memo/i`, OR (c) is annotated with the comment `// PERF-03: per-call compilation allowed because <reason>`. The latter is the escape hatch for cases like the regex worker pool, which compiles user-supplied patterns inside a worker on every call by design.
- Concrete check: file-walk; for each `new RegExp` line, capture the nesting level (count `^\\s*` indent) — at indent 0 it's module scope, at any other indent it must satisfy (b) or (c). Use a simple regex-based file scanner; do NOT add a real parser dependency.
- Known exceptions to whitelist via the `// PERF-03:` comment:
  - src/detect/layer1-regex/worker-pool.ts SINGLE_SHOT_CODE and POOL_WORKER_CODE strings contain `new RegExp(` inside their TEMPLATE STRINGS — these are NOT actual runtime calls, they're worker source code. The grep gate must skip lines inside backtick template literals OR the worker-pool.ts file must opt out via a file-level annotation. Easiest: add file-level annotation `// PERF-03-FILE-EXEMPT: worker-source template literal contains the regex compile; not a hot-path call`.
