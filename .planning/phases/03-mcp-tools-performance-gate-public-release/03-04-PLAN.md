<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Coverage report → quality gate | A silently-disabled threshold would let regressions through |
| Canary-leak grep → audit log | If the grep step is wrong, real leaks pass undetected |
| QA-02 enforcement → integration tests | If the grep can be satisfied without real assertions, the gate is theatre |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04-01 | Repudiation | coverage threshold lowering | mitigate | Task 1 acceptance criterion: thresholds remain at locked 80/80/75/70. Any lowering requires a CONTEXT.md amendment + checkpoint. |
| T-03-04-02 | Information Disclosure | canary-leak gate bypass | mitigate | Two-step gate: in-test assertNoCanaryLeak + workflow-level grep step. A mocked assertNoCanaryLeak would still fail the grep step. |
| T-03-04-03 | Tampering | QA-02 enforcement bypass | mitigate | grep checks tag strings AT the workflow level, not inside the test. A test removing the `@hook-integration` tag fails the workflow even if the test still passes. |
| T-03-04-04 | Denial of Service | matrix slot timeouts | mitigate | timeout-minutes: 15 (test) + 10 (canary-leak); fail-fast: false; concurrency cancellation prevents pile-up. |
| T-03-04-05 | Tampering | v8 ignore comment misuse | mitigate | Each v8 ignore block requires a one-line `// v8 ignore: <reason>` per Task 1 acceptance. Code review surfaces ignore-block additions. |
</threat_model>
