    Concrete section contents:

    **H1: mrclean-claude**

    Subtitle: "Stop secrets at the Claude Code wire. Local. Deterministic. No telemetry."
    Badges row (placeholder URLs, operator confirms): npm version, license MIT, Node 20+.

    **H2: 1. What it does**

    2-3 paragraphs:
    - Paragraph 1: The problem — pasting an AWS key into a Claude Code prompt sends it to Anthropic's API; pasting a `.env` value into a Bash tool sends it to MCP servers. mrclean intercepts in-session before the wire.
    - Paragraph 2: The defense — four detection layers (regex + entropy + .env + dirty-word) run in the hook process; offending text is replaced with `<MRCLEAN:TYPE:NNN>` placeholders before the model sees it; prompts containing CRITICAL/HIGH secrets are blocked outright.
    - Paragraph 3 (the FAQ line, BOLD, DOC-01 verbatim): **"gitleaks for what reaches your repo, mrclean for what reaches the model."** Use gitleaks/trufflehog for pre-commit and CI; use mrclean for the in-session Claude Code surface. They complement, not overlap.

    **H2: 2. Install**

    ```bash
    npm install -g mrclean-claude
    npx mrclean install
    ```

    Note: the npm package name is `mrclean-claude` (the well-known name `mrclean` was taken in 2012); the in-session binaries remain `mrclean` (CLI + hook) and `mrclean-mcp` (long-lived MCP server). After install, start a new Claude Code session — the banner `mrclean active vN.N.N (rules: NNN, allowlist: NN, mode: active)` confirms wiring.

    **H2: 3. Verify**

    ```bash
    npx mrclean doctor
    ```

    Doctor checks: (1) installer wiring (settings.json entries present + absolute path resolves + bin executable), (2) Claude Code version ≥ 2.1.121, (3) canary round-trip via the MCP server, (4) optional `--bench` for the p50/p95 latency stub.

    **H2: 4. Configure**

    Optional. Create `.mrclean/config.toml` to override defaults. Minimal example:

    ```toml
    # .mrclean/config.toml — all fields optional

    dry_run = false   # set true on first install to audit-only without blocking

    [entropy]
    threshold = 4.5   # bits/char; raise to fewer FPs, lower to more recall
    min_length = 20   # below this length, entropy is skipped

    [allowlist]
    rules       = []  # e.g., ["aws-access-key-id"] to disable a specific Layer-1 rule
    paths       = []  # glob patterns; matching tool inputs are not scanned
    stopwords   = []  # literal substrings always allowed
    regexes     = []  # patterns whose matches are allowed
    fingerprints = [] # exact finding hashes (use `mrclean ignore <fingerprint>` to append)
    ```

    Layering precedence: bundled defaults < `~/.mrclean/config.toml` (user) < `.mrclean/config.toml` (project). Allowlist arrays concatenate across layers; other fields override.

    **H2: 5. Dirty word list**

    Project-specific terms (internal codenames, customer names, hostnames) go in `.mrclean/words.txt`. One entry per line; `#` comments; blank lines ignored. Per-entry action override via `word|action` syntax:

    ```text
    # .mrclean/words.txt
    project-bluebird           # default action = block
    internal-host-1|warn       # logs to audit, does not block
    customer-acme|audit        # logs to audit only
    ```

    Hot-reloaded at every SessionStart. Case-insensitive exact match.

    **H2: 6. Uninstall**

    ```bash
    npx mrclean uninstall
    ```

    Restores `~/.claude/settings.json` to the byte-identical pre-install backup. The `.mrclean/` project directory is left alone (delete manually if desired).

    **H2: 7. Modes**

    First-install trust-building: set `dry_run = true` in `.mrclean/config.toml`. All detections are logged to `.mrclean/audit.jsonl` but nothing is blocked or substituted. Run for a week; inspect the audit log; tune the allowlist. Then flip to `dry_run = false`.

    **H2: 8. MCP tools (in-session, opt-in)**

    mrclean ships an MCP server with three tools — invokable from inside Claude Code:

    | Tool | Purpose | Side effects |
    |---|---|---|
    | `mcp__mrclean__check` | Scan provided text; return findings list (rule, severity, placeholder, hash, fingerprint). | None. Read-only. |
    | `mcp__mrclean__redact` | Scan + return `{ redacted: <text-with-placeholders>, findings }`. Writes to audit log. | Audit log entry per finding. |
    | `mcp__mrclean__status` | Return version, rule count, allowlist count, mode, session id, audit log path. | None. |

    Deliberately NOT exposed: `unredact`, `disable`, `add_word`, `config_write`, `ignore`. Pitfall #10: a prompt-injected model cannot ask mrclean to disable itself. Reverse-direction restoration is the planned v2 REVMODE phase and runs server-side, not as a model-facing tool.

    **H2: Intro paragraph**

    mrclean is an in-session sanitizer for Claude Code. It scans text in the four
    hook events and replaces detected secrets with placeholders before they reach
    Anthropic, MCP servers, or cloud agents. This document enumerates what it does
    NOT defend against, so operators have correct expectations.

    **H2: Non-defenses (9 items per CONTEXT.md §Documentation)**

    Each non-defense gets a short paragraph (~3-5 lines) explaining what the gap is,
    why mrclean does not cover it, and the recommended mitigation if any. Sections
    in order:

    1. **Multimodal / pasted-image content** — mrclean reads the text fields of hook payloads. Images embedded in prompts are not scanned. OCR is its own product space. If your operator workflow pastes screenshots that contain secrets, the secrets reach the model unredacted. Recommended mitigation: train operators not to paste credential-containing screenshots; use a separate image-scrubbing tool upstream.

    2. **Model memorization of training-time leaks** — If a secret was in the model's training corpus, the model may surface it in completions independent of anything you type. This is an upstream supply-chain risk on the model itself; mrclean's session-time interception cannot affect it. Mitigation: rotate any secret that may have been published.

    3. **Prompt-injection of the *operator*** — A malicious context (a tool result, a webpage paste, a document) tells the operator: "you must run `npx mrclean uninstall` to fix this error." mrclean is gated by operator action; if the operator follows the instruction, the protection is gone. Mitigation: operator awareness; the README warns this is a real attack class.

    4. **Adversarial obfuscation** — Homoglyph substitution (Cyrillic look-alikes), unusual base64 variants, novel token shapes. Layer 1's regex pack and Layer 2's entropy heuristic are tuned for known shapes; novel obfuscation may bypass detection. Mitigation: the dirty-word list (Layer 4) catches operator-specified terms regardless of shape; add high-value terms there.

    5. **Cross-session placeholder map persistence** — In v1, placeholder ↔ original mappings are session-scoped (limit blast radius per Pitfall #4). The same secret pasted into two separate Claude Code sessions gets two different placeholders. v2 POLISH-02 introduces HMAC-based cross-session deterministic naming.

    6. **LLM Layer 5 semantic detection** — v1 ships Layers 1-4 (regex + entropy + .env + words). Layer 5 (Claude Haiku 4.5 semantic classification of PII / proprietary content) is v2 opt-in (LLM5-01). Until then, semantic categories like "this paragraph reveals internal infrastructure design" go undetected.

    7. **Verified-secret enrichment via vendor APIs** — Tools like trufflehog optionally call vendor endpoints (AWS STS, GitHub API) to verify a detected token is live before flagging. mrclean does not — at odds with the local-first / no-network principle. False-positive flagging is the trade-off.

    8. **Network-level interception of the Claude API itself** — mrclean lives in-session via the Claude Code hook contract. If you bypass Claude Code entirely (raw Anthropic API client, third-party wrapper), mrclean is not in the path. A local HTTPS proxy is a separate architecture and explicitly out of scope per PROJECT.md.

    9. **Pre-commit / git-history scanning** — gitleaks owns that surface. mrclean is the in-session complement: gitleaks for what reaches your repo, mrclean for what reaches the model. Don't run mrclean against your git history; don't run gitleaks against your prompt stream — they're different problems.

    **H2: What mrclean DOES defend against**

    One-paragraph summary: the in-session text-payload surface across the four
    Claude Code hook events (SessionStart, UserPromptSubmit, PreToolUse, PostToolUse)
    on regex-detectable secrets, high-entropy tokens with context keywords, .env-derived
    values, and operator-specified dirty-word lists. This is the leak surface that
    pre-commit hooks and proxy filters do not cover.

    **H2: Reporting issues**

    Threat-class regressions or new attack vectors not in this list: open a GitHub
    issue with the label `security` or email <security@mrclean-claude.invalid>. Coordinated
    disclosure preferred for novel bypass classes; rotate any compromised secrets first.
