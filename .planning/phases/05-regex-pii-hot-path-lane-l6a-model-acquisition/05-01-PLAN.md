<objective>
Ship the Layer 6a regex-PII hot-path lane: a pure-JS synchronous detector that catches
email, US SSN, Luhn-validated credit card, US phone, and IPv4 in-session, behind the
`[pii].enabled` config gate (off by default). PII findings emit in the existing `Finding`
shape with `source: 'pii-regex'` and `ruleId: 'pii:<entity>'`, flowing through the existing
PlaceholderManager, audit log, and 5-axis allowlist with ZERO new sink code.

To avoid duplicating allowlist logic, the private `isAllowlisted` helper in
`layer1-regex/index.ts` is first extracted to a shared `src/detect/allowlist.ts` module
(consumed by both L1 and L6a — and later L6b in Phase 6).

Purpose: Deliver a standalone, model-free PII story that stays inside the v1 perf budget
(current p95 17.4ms vs 100ms budget — ~82ms headroom for 5 compiled patterns).
Output: `src/detect/allowlist.ts`, `src/detect/layer6a-pii.ts`, orchestrator wiring in
`src/detect/index.ts`, plus unit + orchestrator-integration tests.

Implements requirements PII-01 (regex/checksum detection of structured PII) and
PII-02 (existing-shape findings through existing sinks, no new anonymizer/audit/allowlist code).
</objective>

From src/detect/layer1-regex/index.ts (the isAllowlisted helper to EXTRACT — currently private):
```typescript
function isAllowlisted(finding: Finding, config: MrcleanConfig): boolean {
  const al = config.allowlist
  if (al.rules.includes(finding.ruleId)) return true
  if (al.fingerprints.includes(finding.fingerprint)) return true
  if (al.regexes.some((pattern) => { try { return new RegExp(pattern).test(finding.value) } catch { return false } })) return true
  if (al.stopwords.some((sw) => finding.value.includes(sw))) return true
  return false
}
```
