<objective>
Implement `mrclean install` and `mrclean uninstall` end-to-end: atomic JSON edits to `~/.claude/settings.json` (hooks) and `~/.claude.json` (MCP server), absolute-path resolution at install time, idempotency via marker tags, timestamped backups, `.mrclean/` project-directory creation, and a project-root `.gitignore` entry block.

Purpose: This is the operator-facing installer. After this plan completes, an operator can run `mrclean install`, see hook + MCP entries written with absolute paths, run it again with no duplication, and run `mrclean uninstall` to leave the config byte-identical to the backup. Defends against Pitfall #7 (absolute paths) and Pitfall #5 (tmp file in same dir as target). Resolves RESEARCH open question A1 (gitignore lives at project root, not inside `.mrclean/`).

The reader-side counterpart for CFG-01/CFG-03 (project-local config overrides, three-layer precedence) lives in the sibling Plan 01-02b — that plan consumes the `.mrclean/config.toml` stub this plan writes.

Output: Two operator commands that satisfy INST-02..07 and create the `.mrclean/` directory that Plan 01-02b's config reader will read from.
</objective>

Rationale for ignoring `.mrclean/` as a whole directory in Phase 1:
- `audit.jsonl` (Phase 2), `session-*.json`, `manifest-*.jsonl` are all under `.mrclean/` and must never be committed.
- `config.toml` and `words.txt` are the only files an operator might *intentionally* commit. Phase 1 default is "ignore all of `.mrclean/`" — the operator who wants to commit project-local config edits `.gitignore` manually (a one-line change). Surfacing that workflow as a flag is a Phase 2/3 concern.
- The Phase 1 stub `config.toml` is a commented-out template with no live values; Plan 01-02b's config reader treats it as an empty layer, so leaving it gitignored doesn't break CFG-01/CFG-03 — those requirements live behind the reader, not behind whether the stub is committed.
